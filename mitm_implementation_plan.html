Day 2

Add RSA key exchange + AES-GCM encryption (Act 2 base)

01
Understand the key exchange flow before writing any code
Draw this on paper first. Alice and Bob cannot share an AES key securely without first establishing trust. The flow is: both generate RSA keypairs → exchange public keys → Alice encrypts the AES session key with Bob's public key → sends it → Bob decrypts it with his private key → both now share one AES key → all messages encrypted with that key. The MITM attack in Act 2 targets Step 3: swap the public keys before they arrive.
VS Code
The private key NEVER leaves the device. Only public keys travel over the network
The AES session key is 32 random bytes, generated fresh per session by Alice
AES-GCM provides both encryption and authentication — the MITM cannot tamper with a message without the decryption failing
The weakness in Act 2: Alice and Bob trust whatever public key they receive, so the MITM swaps them with his own keys
02
Add encrypted mode to alice.py
When run with the encrypted flag, Alice's startup sequence changes: generate RSA keypair, send public key to server, wait to receive Bob's public key, generate a 32-byte AES key, encrypt it with Bob's RSA public key, send it. Then use AES-GCM for all subsequent messages. No fingerprint check yet.
VS Code
At startup: call generate_rsa_keypair() from crypto_utils. Print: [Alice] Generated RSA keypair
Send a JSON message with "type": "key_exchange", "sender": "alice", "public_key": <PEM string>
Wait (blocking receive) for a reply with "type": "key_exchange" — this is Bob's public key. Deserialize it
Generate aes_key = os.urandom(32). Encrypt it with Bob's received public key using rsa_encrypt()
Send JSON with "type": "session_key", "encrypted_key": <base64 string>
Print: [Alice] Session established. All messages now AES-GCM encrypted.
Chat loop: for each message typed, call aes_encrypt(aes_key, text), add type and sender fields, send the JSON
Receiving thread: on getting a message-type packet, call aes_decrypt() and print the result
03
Add encrypted mode to bob.py
Bob's flow is slightly different: he sends his public key, receives Alice's public key, then waits to receive the AES session key (encrypted with his public key), decrypts it with his private key. Then chats with AES-GCM.
VS Code
Generate RSA keypair. Send public key with "type": "key_exchange", "sender": "bob"
Wait for Alice's public key exchange message. Deserialize and store it (Bob doesn't use Alice's public key for the session key, but stores it)
Wait for the "type": "session_key" message. Decode the base64 ciphertext. Call rsa_decrypt(bob_private_key, ciphertext) → this gives you the 32-byte AES key
Print: [Bob] Received and decrypted session key. Ready.
Chat loop identical to Alice: send/receive using aes_encrypt/aes_decrypt with the shared AES key

Day 3

Act 2 complete — MITM key swap + mitmproxy integration

01
Write mitm.py — the TCP proxy with key-swap logic
This is the most complex file. It's a TCP proxy that listens on port 8080 and forwards to port 9000. The key-swap logic lives here. When it sees a key_exchange message, it replaces the public key with its own. When it sees a session_key message, it decrypts it with its own private key (since Alice encrypted to MITM's key), then re-encrypts with the real recipient's key and forwards. Every message type is logged with its decrypted content.
VS Code
Generate MITM's own RSA keypair at startup: mitm_priv, mitm_pub = generate_rsa_keypair(). Print the fingerprint so you can show it doesn't match Alice/Bob's
TCP server listening on 127.0.0.1:8080. Each accepted connection spawns a thread
Each thread opens its own connection to 127.0.0.1:9000 (the real server)
Per-session state dict: stores alice_real_pub_key, bob_real_pub_key, alice_aes_key, bob_aes_key
On key_exchange from Alice: save her real public key in state. Replace "public_key" field with MITM's serialized public key. Forward the modified message. Print: [MITM] ⚠ Swapped Alice's RSA public key
On key_exchange from Bob (coming back from server): save Bob's real public key. Replace with MITM's key. Forward modified. Print swap notice
On session_key from Alice: decode base64 ciphertext. RSA-decrypt with MITM's private key → get AES session key. Store as alice_aes_key. Re-encrypt the same AES key using Bob's REAL public key. Replace encrypted_key field. Forward
On message from Alice/Bob: AES-GCM decrypt using stored key. Log the plaintext. Forward original ciphertext unchanged
Two threads per session: one reading from client→server direction, one reading server→client direction

Day 4

Act 3 — Fingerprint verification defeats the MITM

01
Add verified mode to alice.py and bob.py
Identical to encrypted mode but with one addition: after receiving the other party's public key, Alice and Bob both display their key fingerprint and the fingerprint of the received key. They then prompt: "Does this match what the other person sees? [yes/no]". If no → abort. This is exactly how WhatsApp Safety Numbers and Signal's safety numbers work.
VS Code
After receiving Bob's public key, Alice calls key_fingerprint(received_key) from crypto_utils
Alice prints a box like:
╔══ FINGERPRINT CHECK ══╗
║ Received key: a1b2c3d4 e5f6a7b8 c9d0e1f2 a3b4c5d6
╚═══════════════════════╝
Prompt: input("Does Bob see the same fingerprint? [yes/no]: ")
If answer is "yes" → continue to session key exchange and chat
If answer is anything else → print ⚠ MITM ATTACK DETECTED. Aborting session. → close socket → exit
Bob has the same check on Alice's key fingerprint
02
Run the Act 3 demo — MITM fails
Run the exact same setup as Act 2 (MITM proxy running). But this time use the verified mode. Alice and Bob will see different fingerprints because MITM swapped the keys. When Alice types "no", the session aborts. MITM cannot read anything.
5 terminals
Terminal 1: python server.py
Terminal 2: python mitm.py — MITM is still running, still trying to intercept
Terminal 3: python alice.py verified
Terminal 4: python bob.py verified
Both Alice and Bob print a fingerprint. Alice's shows MITM's key (because MITM swapped it). Bob's shows MITM's key too — but it's a DIFFERENT MITM key for each side because MITM inserted himself as two separate entities
Type "no" when prompted. Both clients print the red warning and exit
Check MITM's terminal — it shows the key swap happened but the session never completed, so it has zero message content
Screenshot moment 1: Alice's terminal showing the fingerprint mismatch warning — Figure X in report
Screenshot moment 2: MITM's terminal showing it intercepted the key exchange but got no messages — Figure Y

Also run Act 3 WITHOUT MITM — show green path

Run server, alice verified, bob verified. Both print the same fingerprint
Both type "yes". Session proceeds. Chat works
Screenshot: both terminals showing identical fingerprints and green "Verified" message — contrasts clearly with the MITM case
